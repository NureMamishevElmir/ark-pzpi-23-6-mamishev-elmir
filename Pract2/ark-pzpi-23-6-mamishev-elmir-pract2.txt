     У роботі розглядаються сучасні методи рефакторингу, які дозволяють покращувати структуру програмного коду без зміни його зовнішньої поведінки. Рефакторинг є важливою частиною життєвого циклу програмного забезпечення, оскільки забезпечує підвищення якості коду, полегшує його підтримку та робить систему більш гнучкою до майбутніх змін. У межах цієї презентації увагу зосереджено на трьох ключових техніках: Rename Method, Remove Assignments to Parameters та Encapsulate Field. Представлений матеріал дозволяє краще зрозуміти, як системно та безпечно вдосконалювати код, зберігаючи стабільність його функціонування.
     У подальших розділах буде детально висвітлено три методи рефакторингу, кожен з яких вирішує конкретний тип проблем у програмному коді. 


3 ОПИС ВИКОНАНОЇ РОБОТИ
     3.1 Що таке рефакторинг
     Рефакторинг визначається як процес структурного вдосконалення коду без зміни зовнішньої поведінки системи, тобто без впливу на кінцевого користувача чи функціональні можливості програми. Його головна мета полягає в тому, щоб зробити код зрозумілішим для команди, легшим у підтримці та простішим для майбутніх модифікацій. 
     У процесі рефакторингу розробники часто оптимізують структуру класів, прибирають дублювання, найменують сутності більш чітко, покращують інкапсуляцію та усувають приховані залежності. Важливість цього процесу полягає в тому, що програма завжди змінюється: додаються нові функції, змінюється логіка, зростає складність. 
     Без рефакторингу код втрачає якість, стає заплутаним і дорогим у підтримці. Саме тому рефакторинг є фундаментальною практикою професійної розробки.
     3.2 Методи рефакторингу
     Rename Method застосовується для зміни імен методів, коли їх назви не відображають реального призначення, що значно ускладнює розуміння логіки програми. Правильне найменування підвищує читабельність і забезпечує самодокументованість коду. 
     Remove Assignments to Parameters допомагає уникати помилкових ситуацій, коли параметри методу змінюються всередині нього, що може спричиняти плутанину та неочікувану поведінку. 
     Третій метод — Encapsulate Field — дозволяє приховати внутрішні дані класу, замінюючи публічні поля властивостями, які надають контрольований доступ і захищають від некоректних значень. Ці техніки формують фундамент, необхідний для створення надійного та чистого об’єктно-орієнтованого коду.
     3.3 Rename Method
	Метод спрямований на усунення неоднозначних, нерелевантних або застарілих назв методів, які не дають змоги зрозуміти їхню роль та логіку роботи. Невдало підібрані імена створюють бар’єри у сприйнятті коду, ускладнюють пошук помилок та збільшують час, необхідний на підтримку. Правильне перейменування дозволяє зробити код самодокументованим: коли ім’я методу точно відображає його дію, відпадає потреба у зайвих коментарях або детальному аналізі внутрішньої реалізації. Такий рефакторинг особливо корисний під час робіт над спадковими системами, де логіка сформована раніше і новим розробникам важко швидко розібратися в структуруванні. Завдяки інструментам IDE метод можна застосувати без ризику порушення зв’язків між частинами програми.
     Приклад на C#:
1 // До рефакторингу
2 public decimal GetP(int custId) 
3 { 
4     return orders.Where(o => o.CustomerId == custId)
5     .Sum(o => o.Total);
6 }
7 
8 // Після рефакторингу
9 public decimal CalculateCustomerTotalRevenue(int customerId) 
10 { 
11     return orders.Where(o => o.CustomerId == customerId) 
12     .Sum(o => o.Total);
13 }
     3.4 Remove Assignments to Parameters
     Цей метод вирішує проблему, пов’язану зі зміною вхідних параметрів усередині методу. Хоча у C# параметри передаються за значенням, сам факт їх перезапису всередині методу створює ризик неправильного сприйняття логіки.
      Розробник, читаючи код, очікує, що параметр відображає те значення, яке було передано у виклику, а не нове, модифіковане. Натомість використання окремих локальних змінних робить поведінку методу передбачуваною і підвищує зрозумілість. 
     Це покращує структуру функціонального блоку та мінімізує можливість прихованих помилок, що можуть виникати під час зміни параметрів. Apply Discount у прикладі демонструє, як просте розділення параметра і змінної робить логіку очевиднішою та чистішою.
     Приклад на C#:
1 // До рефакторингу
2 public int ApplyDiscount(int price) 
3 { 
4     price = price * 0.9;
5     return price;
6 }
7 
8 // Після рефакторингу
9 public int ApplyDiscount(int price) { 
10 int discountedPrice = price * 0.9; 
11 return discountedPrice;
12 }
     3.5 Encapsulate Field
     Метод Encapsulate Field вирішує класичну проблему порушення інкапсуляції, коли поля класу є публічними та доступні для зміни будь-звідки. Така модель створює ризик порушення цілісності об’єкта, оскільки значення можуть бути змінені неконтрольовано або у неправильному контексті. Інкапсуляція через властивості дозволяє контролювати зміну внутрішнього стану, додаючи логіку перевірки, фільтрації або навіть автоматичного коригування даних. Наприклад, поле віку, яке не може бути від’ємним, обмежується у сеттері, що забезпечує захист об’єкта від некоректних значень. У результаті клас стає більш стійким до помилок, а його поведінка — більш передбачуваною.
     Приклад на C#:
1 // До рефакторингу
2 public class Customer
3 {    
4     public string Name;
5     public int Age;
6     public void UpdateInfo() 
7     {
8     // Прямий доступ до полів        
9         Name = "Новий клієнт";
10         Age = -5; // Помилка!
11     }
12 }
13 
14 // Після рефакторингу
15 public class Customer
16 { 
17     private string name; 
18     private int age;
19     public string Name 
20     {
21         get { return name; } 
22         set { name = value; } 
23     }  
24     public int Age 
25     {
26         get { return age; } 
27         set {  if (value >= 0) age = value; } 
28     }
29 }
     3.6 Інструменти для рефакторингу
     У процесі рефакторингу важливу роль відіграють інструменти, які дозволяють автоматизувати і спростити більшість рутинних операцій. 
     Visual Studio пропонує широкий набір вбудованих функцій, таких як перейменування, інкапсуляція полів, виділення методів та пошук використань, що значно прискорює роботу і зменшує ймовірність помилок. 
     JetBrains Rider забезпечує ще глибший аналіз коду, пропонуючи оптимізації, підказки і автоматичне виявлення проблемних місць, що робить його сильним інструментом для великих проєктів. 
     ReSharper, як розширення для Visual Studio, додає сотні додаткових інспекцій і рефакторингів, допомагаючи підтримувати високу якість коду протягом усього циклу розробки. 
     Використання цих інструментів дозволяє розробникам зосередитися на логіці, а не на технічних деталях переписування коду, підвищуючи продуктивність та ефективність роботи команди.
     3.7 Коли застосовувати рефакторинг
     Рефакторинг варто виконувати регулярно. Перед додаванням нової функціональності рефакторинг допомагає очистити та впорядкувати код, підготувавши його до розширення таким чином, щоб нові зміни не збільшували хаос у структурі проєкту. 
     Під час виправлення помилок рефакторинг дозволяє знайти глибші причини несправностей, а також спростити місця, які важко зрозуміти або перевантажені логікою. На етапі код-рев’ю рефакторинг стає інструментом покращення якості, оскільки дає змогу виявити та виправити проблемні місця, які важко помітити під час звичайної роботи. 
     Регулярна підтримка коду за допомогою рефакторингу допомагає уникати накопичення технічного боргу і зберігати стабільність системи у довгостроковій перспективі.


ВИСНОВКИ
     Хоча на перший погляд може здаватися, що на рефакторинг витрачається час без додавання нових функцій, насправді він значно скорочує витрати на підтримку і пришвидшує розробку у майбутньому. 
     Систематичне застосування методів рефакторингу — таких як Rename Method, Remove Assignments to Parameters та Encapsulate Field — дозволяє підтримувати код у чистому й впорядкованому стані, полегшує взаємодію між членами команди та зменшує кількість помилок. 
     Сучасні інструменти розробки автоматизують більшість цих процесів, що робить їх швидкими, надійними та безпечними. 
     Таким чином, рефакторинг стає обов’язковою складовою професійної практики програмістів, які прагнуть створювати якісні та стійкі системи.
