2 ЗАВДАННЯ
     Метою цієї роботи є вивчення офіційних рекомендацій щодо оформлення програмного коду мовою C# та демонстрація їх практичного застосування відповідно до Google C# Style Guide та .NET Coding Conventions. Студент має проаналізувати правила стилістичного оформлення, визначити їх ключові принципи та показати, як саме ці рекомендації впливають на якість, читабельність та підтримуваність програмного коду.
Під час виконання роботи необхідно розглянути основні аспекти стилю:
– правила іменування файлів, класів, інтерфейсів, методів, змінних і приватних полів;
– вимоги до форматування коду (відступи, пробіли, довжина рядка, використання фігурних дужок);
– структурування класів та методів, логічний поділ функціональності;
– використання характерних для C# ідіом та підходів;
– принципи створення чистого, однорідного та легкого в підтримці коду.
     До кожного правила необхідно підготувати приклади правильного та неправильного оформлення, супроводивши їх поясненнями, чому певний принцип є важливим і як він допомагає уникати помилок у майбутньому. стилю для командної роботи, стабільності та масштабованості проєктів.


3 ОПИС ВИКОНАНОЇ РОБОТИ
     3.1 Вибір мови програмування та джерел
     C# — це сучасна об’єктно-орієнтована мова програмування, яка займає центральне місце у світі .NET розробки. Вона використовується для створення веб-додатків, серверних API, настільних програм, мобільних застосунків та навіть великих ігрових проєктів. Гнучкість і масштабованість цієї мови дозволяють розробникам вирішувати широкий спектр задач, але саме через це у великих командах часто виникає проблема нестабільності стилю.
     Коли кожен пише код «по-своєму», виникають труднощі зі спільним розумінням структури, логіки та стандартів. Це призводить до накопичення технічного боргу, ускладнює підтримку, рев’ю та оновлення системи. Саме тому в професійних командах використовують правила оформлення коду — code conventions. Вони встановлюють єдині вимоги до стилю, структури та назв, забезпечують передбачуваність та зменшують кількість помилок.
     Google C# Style Guide — це набір узгоджених практик, який поєднує досвід Microsoft і підхід бібліотек CoreFX, забезпечуючи розробникам чітку базу для створення охайного, читабельного та підтримуваного коду.
     3.2 Важливість стилю кодування
     Стиль кодування — це фундамент роботи будь-якої команди. Код — це текст, який пишеться один раз, але читається десятки або сотні разів іншими членами команди або навіть самим автором через тривалий час. Якщо стиль не узгоджений, кожен новий модуль виглядає по-іншому, що створює додаткову когнітивну складність.
     Google C# Style Guide систематизує правила, вироблені галуззю, і допомагає уникати хаосу у проєктах. Він забезпечує:
• сталість стилю, незалежно від того, скільки людей працює над кодом;
• прозорість логіки, коли структура та іменування не викликають запитань.
• прискорення комунікації, оскільки всі розробники «говорять однією мовою»;
• зменшення помилок, які виникають через різні стилістичні підходи.
   3.3 Основні принципи
   1. Покращена читабельність
     Код, оформлений за правилами, завжди виглядає однаково: предбачувані відступи, сталі іменування, зрозумілий порядок методів. Це допомагає швидше орієнтуватися у проєкті, знаходити помилки й робити зміни без зайвих затримок.
   2. Єдність у команді
     Коли всі користуються одним Style Guide, кожен модуль виглядає як частина єдиного цілого. Різниця між стилями окремих розробників зникає, а це значно полегшує командну взаємодію.
   3. Легка інтеграція нових членів
     Новим людям простіше почати працювати над проєктом, якщо структура кодової бази зрозуміла і всі правила визначені заздалегідь. Це скорочує час адаптації та дозволяє швидко долучатися до розробки функціональності.
     3.4 Правила іменування
     Згідно з Google C# Style Guide, правила іменування визначають чіткий та послідовний стиль оформлення коду. Простори імен, класи, методи та публічні властивості оформлюються у стилі PascalCase. Цей стиль починається з великої літери та робить назву чіткою й видимою серед інших елементів. Це дозволяє з першого погляду визначити тип сутності, її рівень доступу та роль у системі.
1 // Поганий приклад
2 public class exceptionResponse
3 {
4 [JsonProperty("error")]
5 public string error { get; set; }
6 }
7 
8 // Гарний приклад
9 public class ExceptionResponse
10 {
11 [JsonProperty("error")]
12 public string Error { get; set; }
13 }
     Локальні змінні й параметри методів використовують стиль camelCase  — це дає змогу відрізняти локальний контекст від глобального та структурованого.
1 // Поганий приклад
2 public ConflictException(string Message)
3 {
4 }
5 
6 // Гарний приклад
7 public ConflictException(string message)
8 {
9 }
     А приватні поля — формат _camelCase. Підкреслення _ відразу сигналізує, що це внутрішній елемент об'єкта.
1 //Поганий приклад
2 private readonly ITokenGenerator _TokenGenerator;
3 private readonly JwtSettings _JwtSettings;
4 private readonly ITokenRepository _TokenRepository;
5 
6 // Гарний приклад
7 private readonly ITokenGenerator _tokenGenerator;
8 private readonly JwtSettings _jwtSettings;
9 private readonly ITokenRepository _tokenRepository;

     Інтерфейси традиційно починаються з літери І. Це стандарт C#, який підтримує зрозумілу структуру й дозволяє легше ідентифікувати сутності за призначенням.
1 // Поганий приклад
2 public interface ApplicationManager
3 {
4 }
5 // Гарний приклад
6 public interface IApplicationManager
7 {
8 }
     3.5 Додаткові правила іменування
     Імена файлів повинні бути в PascalCase та відповідати назві основного класу, який вони містять (наприклад, ApplicationManager.cs). Уникайте використання скорочень; імена мають бути чіткими та зрозумілими.
     Модифікатори (такі як const, static, readonly) не впливають на стиль іменування. Завжди дотримуйтесь встановлених правил іменування, незалежно від модифікаторів.
     Акроніми слід писати повністю в PascalCase, наприклад, ApplicationManager, а не ApplicationMANAGER, для збереження однорідності.
1 // Поганий приклад
2 // applicationManager.cs
3 public class applicationManager
4 {
5 }
6 
7 // Гарний приклад
8 // ApplicationManager.cs
9 public class ApplicationManager
10 {
11 }
     3.6 Форматування коду
     Відступи: Використовуйте 4 пробіли для відступів; табуляції суворо заборонені.
     Довжина рядка: Максимальна довжина рядка не повинна перевищувати 100 символів.
     Пробіли: Ставте пробіли після ключових слів (if, for, while) та після ком, але не всередині дужок.
     Обов'язкові дужки: Завжди використовуйте фігурні дужки для блоків коду, навіть якщо вони містять лише один оператор, щоб уникнути неоднозначностей.
1 // Поганий приклад
2 if(string.IsNullOrWhiteSpace(clientIdString))
3 {
4 }
5 
6 // Гарний приклад
7 if (string.IsNullOrWhiteSpace(clientIdString))
8 {
9 }
     3.7 Коментарі
     Коментування — частина культури коду. Google Style Guide визначає:
     Короткі коментарі. Використовуються через //, коли потрібно уточнити невеликий фрагмент.
     Блокові коментарі. Використовуються через /* ... */ для пояснення складних або важливих частин логіки.
     Коментар має пояснювати «чому».
     Це зменшує ризик неправильного використання коду в майбутньому.
1 // Поганий приклад
2 /// <summary>
3 /// Order status after refund
4 /// </summary>
5 public string Status { get; set; }
6 
7 /// <summary>
8 /// Order status
9 /// </summary>
10 public string Status { get; set; }
     3.8 Структура класів і методів
     	Групування методів. Методи повинні бути впорядковані за призначенням: робота з даними, бізнес-логіка, валідація, допоміжні операції. Це полегшує навігацію по файлу та пришвидшує розуміння структури класу.
     Метод — одна відповідальність. Кожен метод має виконувати одну чітко визначену дію. Це підвищує читабельність, спрощує тестування та зменшує ризик побічних ефектів.
     Уникати дублювання. Повторювані фрагменти коду варто винести в окремі методи або сервіси. Це знижує кількість помилок, спрощує подальші зміни та підтримку застосунку.
1 // Поганий приклад
2 public class UserService
3 {
4     public bool RegisterUser(string name, string email)
5     {
6         if (!email.Contains("@"))
7             return false;
8 
9         var db = new Database();
10         db.Save(name, email);
11 
12         return true;
13     }
14 }
15 // Гарний приклад
16 public class UserService
17 {
18     public bool RegisterUser(string name, string email)
19     {
20         if (!email.Contains("@"))
21             return false;
22 
23         var db = new Database();
24         db.Save(name, email);
25 
26         return true;
27     }
28 }


ВИСНОВКИ

	Дотримання правил оформлення програмного коду за Google C# Style Guide є ключовим фактором у створенні якісного, чистого та передбачуваного коду. Уніфікований стиль підвищує читабельність, спрощує командну роботу та пришвидшує адаптацію нових розробників у проєкті.
	Правильне іменування, чітке форматування, структурованість та коректне коментування коду мінімізують ризик помилок і забезпечують довготривалу підтримуваність програмного продукту.
	Єдиний стиль — це не формальність, а практичний інструмент, який значно підвищує ефективність розробки. Він дозволяє зосередитися на логіці та функціональності, а не на розумінні чужого стилю написання.
	Використання узгоджених конвенцій, таких як Google C# Style Guide, робить код впорядкованим, масштабованим та більш професійним — що є критично важливим для сучасних команд і великих проєктів.
